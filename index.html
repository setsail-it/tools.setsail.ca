Workflow Guide
Interface

Left sidebar — shows all 11 pipeline stages. Green checkmark = done. Active stage highlighted dark. Click any stage to jump directly — no stage locking, all stages are always accessible.

Top bar centre — AI bar — four modes: (1) Generating: indeterminate pulse + elapsed timer + Stop button while a Claude call is running. (2) Stopped: shows Resume and Dismiss buttons when a bulk operation is paused — Resume continues from exactly where it stopped. (3) Notification: success or error message after an action (hover to see full text if truncated). (4) Tips: stage-specific workflow tips rotate every 7s when idle — pauses during generation, resumes after.

Project switcher — click the project name in the top bar to switch between projects without going back to the projects screen.

Admin panel — click your name (top right) → Admin. Manage team members: name, email, role (Admin / Strategist / Viewer), and project access (specific project IDs or blank = all). Edit your own name from the same panel. Log out via the same dropdown.

Button tooltips — hover any action button for 0.3s to see a description of exactly what it does. Implemented across Keywords, Briefs, and Sitemap stages.

Pipeline Overview
S1Setup — client name, URL, primary market, industry, discovery notes, sales qualification, strategy doc upload
S2Snapshot — domain authority, backlinks, referring domains, top organic pages, competitor DA comparison
S3Research — AI enriches 150+ fields across 5 tabs, client pain extraction, unit economics, completeness scoring with inline tab percentages
S4Strategy — 8 AI diagnostics (D0–D7), scoring engine, positioning direction, audience intelligence, Keywords tab (AEO pipeline), demand validation, web strategy brief
S5Sitemap — page architecture, keyword mapping, page goals, persona/voice/positioning, mermaid + HTML export
S6Briefs — niche expand → assign keywords + questions → generate → evaluate → approve, with queue support
S7Copy — generate from brief → multi-pass audit → human QC → approve, with positioning/persona/pain injection
S8Images — content-aware visual planning: auto-slots from copy structure, visual type detection, persona/emotional targeting, per-slot approval workflow
S9Layout — wireframe generation with section-level structure, responsive grid, component assignment per page
S10Schema — FAQ + LocalBusiness + Service JSON-LD from approved copy and research data
S11Export — package approved copy, images, schema, and layout for development handoff
Stage 1 — Setup
·Client name + website URL — used in all downstream copy, SEO output, and domain lookups. Fill carefully.
·Primary Market — the default geo for all pages. Drives country auto-detection for keyword volumes ("Vancouver, BC" → Canada, "Sydney" → Australia). Override per page in the Sitemap Market column for multi-market projects.
·Industry — sets the vertical context for Research enrichment, Strategy diagnostics, and downstream content.
·Strategy document upload — paste or upload your client strategy doc. This feeds Research enrichment and Strategy generation. The website strategy brief is auto-generated in the Strategy stage (Output tab) after diagnostics complete.
·Discovery notes — paste call notes, client conversations, or any qualitative context. These feed directly into Strategy D0 (Audience) and positioning hypothesis extraction.
·Sales qualification / sales notes — budget range, decision timeline, deal stage. Feeds Unit Economics (D1) and channel budget allocation (D4).
·Competitor URLs — up to 5 direct competitors. Used in Snapshot comparison, Research competitor tab, and Strategy positioning analysis.
·Fields like voice, pricing, and image style are derived automatically in Research (Stage 3) — they are not entered here. Setup is intentionally minimal.
Stage 2 — Site Snapshot
·Pulls domain metrics via DataForSEO — Domain Rating, backlink count, referring domains, organic traffic estimate, and top organic pages.
·Top organic pages from the snapshot are auto-imported into the Sitemap stage as Active Pages for consideration.
·Competitor URLs from Setup are compared side-by-side — DA comparison helps set realistic ranking expectations.
·The DR from this snapshot feeds into the Site DR field in the Keywords Opportunities tab for rankability scoring.
·Snapshot data is stored in S.snapshot and saves with the project. Re-run only if domain metrics have changed significantly.
Stage 3 — Research · Best Order

AI extracts 150+ structured fields from your strategy docs across 5 tabs. Every field feeds downstream — briefs, copy, schema, and layout all pull from Research. Per-tab completion percentages show inline next to each tab name; a compact progress bar at top shows overall completeness.

  1. Upload client documents — strategy doc, intake form, existing content, or anything with context. Strategy docs up to 12,000 chars are fully parsed (reference docs up to 10,000 chars each).
  2. Enrich each tab — Business, Audience, Brand, Schema & Local, Competitors. Or use "Enrich with AI" to run all 5 sequentially.
  3. Business tab — core identity, services, pricing. The Services (Detailed) repeater captures per-service descriptions, target audiences, and differentiators — these feed brief and copy context.
  4. Audience tab — buyer profile, pain points, objections, sales process, unit economics inputs (budget, deal size, LTV, lead quality, close rate). CTA Architecture section defines primary, secondary, and low-commitment CTAs — injected into every brief and copy prompt.
  5. Audience tab: Client Pain — pain points, triggers, language, severity ratings, and urgency signals extracted from strategy docs. This is Layer 1 of the dual pain architecture. The scorecard shows a warning badge when pain data is incomplete. Pain context feeds all downstream prompts via getPainContextBlock().
  6. Brand tab — voice, differentiators, visual assets, and the Proof & E-E-A-T section: case studies, notable clients, awards/certifications, team credentials, founder bio, publications. These feed copy Pass 3 (E-E-A-T injection) and are included in all brief prompts.
  7. Schema & Local tab — business type, address, social profiles, FAQs, reviews. All injected into the Schema stage prompt for accurate JSON-LD generation.
  8. Competitors tab — name, URL, strengths, weaknesses, and what we do better. Competitor weaknesses appear in brief context so copy can address competitive gaps.
  9. Re-Enrich — the Re-enrich button bypasses skip logic and overwrites all fields, even pre-filled ones.
⚠ Do not pre-fill default values in research fields before enrichment — the skip logic treats any non-empty string as "already filled" and will not overwrite it.
Stage 4 — Strategy

The strategy engine runs 8 AI diagnostics (D0–D7), scores each on three dimensions, and produces a comprehensive strategy. It includes a Keywords tab for demand validation, positioning direction system, audience intelligence, and a compiled strategy document.

·D0 Audience runs first — segments, personas (archetype labels), buying motions, purchase triggers, objection maps. Persona executionProfiles include emotional drivers, competitor frustrations, and switching triggers (Layer 2 of dual pain architecture).
·D1–D7 run sequentially: Unit Economics, Competitive Position, Subtraction, Channel Viability, Website & CRO, Content & Authority, Risk Assessment.
·Positioning Direction — enter founder hypotheses → Evaluate against competitors → Select a direction. All downstream output is constrained by the selected direction.
·Scoring engine — Data Completeness (35%), Analytical Confidence (40%), Specificity (25%). Anti-inflation caps prevent score gaming. See the Strategy Scoring Engine section below for full details.
·Web Strategy Brief (Output tab) — auto-generated after diagnostics complete. 7-section brief injected into every brief, copy, and sitemap generation call downstream.
·Customer Voice enrichment — enriches each persona with executionProfile data (emotional drivers, competitor frustrations, switching triggers, decision anxiety). Combined with Layer 1 client pain via getPainContextBlock() for all downstream prompts.
Keywords Tab · AEO Pipeline
The Keywords tab within Strategy follows an AEO (Answer Engine Optimisation) pipeline: Questions → Seeds → Opportunities → Clusters. Each sub-tab feeds the next.
  1. Questions tab first — click Fetch Questions to generate 20 buyer-intent questions from Research data (services_detail, case studies, competitors, pricing model, pain points). Intent keywords are auto-extracted into Seeds so you do not need to manually pull them. Click More Questions for another 20 (deduped). Click Blog Seeds to convert informational questions into blog keyword seeds.
  2. Validate & Assign — checks DataForSEO volumes for each question, then AI assigns each to a page (service FAQ, new blog, or sitewide FAQ). Coverage column appears after clustering — gap questions get a “+ Sitemap” button. This mapping flows into the cluster prompt so AEO page assignments stay aligned.
  3. Set Country in the Seeds tab dropdown — auto-detects from geo text (AU/CA/US/GB/NZ/SG/ZA). Override manually if wrong. Wrong country = wrong volumes.
  4. AI Generate Seeds — produces 20-30 tight head terms (2-4 words) from Research context. Competitor Keywords mines non-brand organic terms from competitor domains. Mechanical Seeds rebuilds the service x geo matrix from Setup data.
  5. Seed bucket filter bar — four sources: Mechanical, AI, Competitor, Questions. Each shows a count. Verify all relevant buckets are populated before Fetch Volumes.
  6. Fetch Volumes — Google Suggest expands each seed into 400-600 real phrases, then DataForSEO volumes each one. Top 300 by score are kept. This is your full keyword pool.
  7. Enter Site DR in the Opportunities tab — enables rankability badges per keyword (green = rankable within DR range, yellow = stretch, red = too competitive).
  8. Review Opportunities — scored by log(vol+1) x 100 / max(KD, 5). Select Top 50 then manually add strategic low-volume terms (high-CPC, direct buyer intent, core services).
  9. Cluster Selected — AI groups keywords into page clusters using full research context: value proposition, differentiators, services_detail, competitor weaknesses, audience, and buyer roles. Clusters use page-title naming (e.g. “SEO Services Vancouver”).
  10. Strategic Override — core services from services_detail always qualify for pages even with volume under 50. The AI will not disqualify your flagship services.
  11. Blog Architecture — clusters prefer 3-6 substantial blog pillars mapped to core services over 10+ thin standalone posts. Questions assigned as “new_blog” join or create blog clusters.
  12. Build Sitemap from Clusters — each cluster becomes a page in Stage 5. Review cluster names and anchors before proceeding.
⚠ Run Fetch Volumes before building the Sitemap. Pages will not have volume data otherwise and the sitemap generator will not be able to justify page creation thresholds.
Stage 5 — Sitemap · Page Rules
Two types of pages — Claude enforces volume thresholds for SEO pages:
StructuralHomepage, /about, /contact, /case-studies — always built, vol:0 acceptable
/services/vol > 0 required — any keyword cluster justifies creation
/industries/vol > 100 required
/locations/vol > 100 required
/blog/vol > 50 required
Keyword intent rules (enforced in prompts):
·Homepage — brand/category only: "Vancouver marketing agency". NEVER specific service terms.
·Service pages — service-specific only: "SEO services Vancouver", "PPC agency BC".
·Location pages — geo + category: "digital marketing Calgary", "marketing agency Toronto".
  1. Generate Sitemap — Claude maps pages to keyword clusters and assigns a page goal (strategic purpose) to each page based on the website strategy. When a positioning direction is selected, it is injected into the AI prompt so page architecture reflects the chosen strategic angle.
  2. Display order — pages are automatically sorted: Core (home, about, contact) → Services → Locations → Blog. Within each type: P1 → P2 → P3 → score. Display-only — does not change generation priority.
  3. Category tabs — filter by All, Services, Locations, Blog, or Core. Counts update live.
  4. Grid columns — #, Page, Cluster Anchor, Vol, KD, Score, Intent, Priority, Market, Traffic. Intent and Market are editable in Edit mode.
  5. Page Goal (purple text under page name) — the strategic purpose of the page. AI-generated from strategy + page type + keyword intent + persona context. Click the sparkle button per page or use Goals toolbar button to bulk-generate for all pages without one. Editable in Edit mode. Flows into brief and copy prompts.
  6. Persona badge (blue badge after content pillar) — shows the target persona assigned to each page. Auto-assigned during Build based on page type and audience segments. Editable via dropdown in Edit mode. Persona context (frustrations, decision criteria, language patterns) is injected into page goal and brief prompts.
  7. Voice overlay — per-page voice rule assignment. Auto-assigned from the persona segment name, defaults to "base". Editable via dropdown in Edit mode. Briefs stage reads the overlay and injects vertical-specific voice rules from strategy into the brief prompt.
  8. Market column — per-page target geo. Leave blank to inherit Primary Market from Setup. Override for multi-market pages (e.g. US landing pages on a Canadian project). Keyword volume Refresh groups lookups by target market.
  9. Intent column — search intent classification (commercial, transactional, informational, navigational). Colour-coded. Editable dropdown in Edit mode.
  10. Priority + budget awareness — AI-suggested priority (P1/P2/P3) factors in alignment score, page type, and budget tier. If a lever is unfunded in the current budget tier, the page is capped at P3 with a tooltip explaining the budget threshold needed.
  11. Stats bar strategy badges — when strategy data exists, the stats bar shows: alignment score average, priority distribution diff, positioning direction gaps (purple badge — missing proof pages), and CTA landing page gaps (orange badge — CTAs without dedicated pages).
  12. Persona Coverage Check — panel in the Content Intelligence section. Verifies each active persona has an industry page, case study, and objection-addressing content. Missing items show "+ Create" buttons to add gap-filling pages directly.
  13. Refresh — re-fetches keyword volumes from DataForSEO, grouped by each page target market. Multi-country projects get separate API calls per geo.
  14. Mermaid Export — exports the sitemap as a Mermaid diagram. Open the modal, click Copy to Clipboard, then paste in FigJam (Insert → Mermaid diagram → paste → Done).
  15. HTML Sitemap Export — opens a styled HTML sitemap in a new tab showing all pages grouped by priority with URL, primary keyword, and copy/schema/images completion status.
  16. Approve Sitemap — locks the page architecture. Required before Briefs stage processes.
Stage 6 — Briefs · Best Order
  1. Expand Niche Keywords — runs Google Suggest on each page's primary keyword with 10 modifiers. Pulls niche variants and gets DataForSEO volumes. Tagged by page slug so AI Assign knows which keywords belong where. Run globally from the toolbar, or per-page via the ⌖ Niche KW button on each brief card.
  2. Generate Page Questions — one Claude call generates 8 targeted FAQs per page based on primary keyword + page type. Auto-assigned to each page's assignedQuestions[]. Run globally or per-page via the ? Questions button on each brief card.
  3. AI Assign Keywords + Questions — distributes remaining global PAA questions and keyword pool across pages. Skips already-assigned page questions (no duplicates). Run globally from the toolbar. The per-page ✦ Assign button on each brief card runs a full curation pass: Claude reviews what's currently assigned, keeps what matches the page's intent, drops what doesn't, and replaces it with better fits from the pool. The button shows a removal count (e.g. ✓ Curated (3 removed)) when items were dropped.
  4. SERP Intel (per page, or auto-runs at brief generation) — fetches top 3 organic results for the primary keyword, extracts: word count, H2 structure, KW density. Sets word count target at max competitor × 1.05. Some sites block crawlers — this is normal.
  5. Generate / Regenerate Brief — click any page row to expand it (accordion). Once open, use the top-right action bar to generate, regenerate, or run a new version. SERP Intel auto-runs silently if not already done.
  6. Queue All — submits all un-generated briefs to the Cloudflare Queue for server-side generation. Tab-close safe — the worker generates briefs in the background even if you leave. Processes 2 concurrent, retries automatically on rate limits. Progress shows in the AI bar with estimated time remaining. Results merge back into the page when done.
  7. Stop / Resume — click Stop in the AI bar to pause any generation mid-stream. The Resume button appears and picks up exactly where it left off. Works for Write All Briefs, Enrich All, page goals, and queue polling.
  8. Pre-generation validation — warnings appear in the AI bar if a page is missing its primary keyword, assigned questions, business overview, or primary CTA. Generation still runs but the brief quality will be lower.
  9. Evaluate Brief — AI scorecard: 9–11 checks including reader profile, H1, CTA presence, objections, trust signals, SERP word count target, H2 coverage, and page goal alignment.
  10. Version pills (V1 / V2…) — appear in the score bar once more than one draft exists. Click to switch between versions. Last 2 kept — V1 is dropped when V3 is pushed. Run V2 via + V2 button in the action bar.
  11. Approve Brief — button in the top-right action bar of each card. Locks the brief. Copy stage will not process a page without an approved brief.
✓ Steps 1–3 take about 3 minutes total and dramatically improve brief specificity — especially for niche service/industry pages that previously got generic questions and thin keyword coverage.
Stage 7 — Copy · Best Order
  1. Brief must be approved before copy generates for that page. The Generate button is locked otherwise.
  2. Generate Copy (Pass 1) — full page HTML from brief. Stored as draft v1. Copy HTML is saved to its own KV key (not the main project payload) so it survives page reload.
  3. Run Audit — deterministic checks first (keyword density 0.3–3.0%, SERP-calibrated ceiling if SERP Intel ran), then AI checks: H1 contains keyword, primary keyword in intro, intent match, no placeholders, word count vs target, Canadian spelling, no AI fluff. Page-type specific checks added on top.
  4. Pass 2 — Claude surgically rewrites weak sections based on audit findings. Generates a new draft tab without replacing v1.
  5. Human QC — 7 items per page (6 universal + 1 page-type specific):
    • Stats & numbers verified (every %, $, count confirmed against real source)
    • Client proof approved for use (named clients/logos have permission)
    • Offer & pricing current (guarantees match current offering)
    • CTAs & links work (no 404s or dead forms)
    • Brand voice on-point (sounds like us, not generic AI)
    • No stale content (no outdated team refs, discontinued services)
    • + 1 page-type specific item
  6. Pain sidebar — expanded copy panel shows a collapsible pain context sidebar with both Layer 1 (client pain from Research) and Layer 2 (customer pain from persona executionProfile). Helps the reviewer verify pain language is woven into the copy.
  7. Approve Copy — locks the page for export and Schema generation. Last 2 copy drafts kept per page.
⚠ Draft version tabs (v1/v2/v3) are in-memory only. Only the active draft persists across a page reload. Version history is lost on refresh — do not rely on it for long-term storage.
Stage 8 — Images · Content-Aware Visual Planning

Reads approved copy HTML to auto-generate image slots per page section. Each slot gets a visual type, source route, persona targeting, and emotional tone — not just "5 generic photos."

  1. Auto-slot generation — parses copy HTML (H1/H2/H3 sections) via TreeWalker. Creates a hero slot, one slot per major section break, extras for long sections (400+ words), and a CTA slot. Slot count varies per page based on actual content.
  2. Visual type detection — regex matches section headings and body text to assign types: photography, photography_people, illustration, diagram, data_viz, social_proof, icon, before_after, screenshot. Each type auto-routes to a source.
  3. Source routing — ai_generate (Gemini), upload (file picker with WebP compression), brief_only (written image brief instead of image), placeholder (generates a placeholder visual with a note that real asset is needed). Override per slot via dropdown.
  4. Persona and emotional targeting — each slot inherits the page target persona from Sitemap. Journey stage maps to emotional tone: awareness → curiosity, consideration → confidence, decision → urgency. Persona executionProfile emotional drivers are injected into prompts.
  5. Competitor voice avoidance — prompts include anti-patterns from competitor brand analysis so generated images do not look like competitor visual language.
  6. Approval workflow — per-slot status: pending → approved / redo / replace / removed. Redo clears the image for re-generation. Replace switches source to upload. Removed soft-deletes (restorable). Only approved slots carry forward.
  7. Re-analyse from Copy — when copy changes, click the refresh button per page to rebuild slots from the new copy structure. Existing images are preserved by label matching — only new/removed sections change.
  8. Copy preview panel — collapsible panel per page shows parsed section structure with visual type assignments. Useful for verifying slot-to-section mapping.
  9. Bulk generate — generates all pages or per-page. Skips approved and removed slots. Stop/Resume works mid-generation with exact position tracking.
  10. Upload — file picker for social_proof, before/after, screenshots, or any slot you want a real image for. Auto-compresses to WebP. Saved to per-slug KV key.
  11. Custom prompts — click any slot to edit the AI prompt before generation. Useful for specific creative direction.
  12. Backward compatible — existing projects with the old 5-slot format get new fields merged via mergeSlotDefaults(). No data loss on upgrade.
⚠ Image data is stored in separate KV keys per page (u:{email}:img:{projectId}:{slug}). Both scheduleSave() and savePageImages(slug) must be called for changes to persist.
Strategy Scoring Engine

Every strategy section is scored on three dimensions. The weighted average produces a section score (0–10), and section scores roll up into an overall strategy score.

Three Scoring Dimensions
Data Completeness (35%)Are all required inputs present? Each section defines a checklist of fields — missing fields reduce this score proportionally.
Analytical Confidence (40%)How trustworthy is the analysis? Derived from content depth (key count in the section data) and audit pass rate. Sections with 8+ keys score 7, 4+ keys score 5.5, fewer score 4. High audit pass rate (≥90%) adds +1, low pass rate (<50%) deducts -1.5.
Specificity (25%)Is the output specific to this client? Generic outputs score low. Concrete data — real numbers, named competitors, validated differentiators — scores high.
Section Weights
·Positioning — 18%  |  Channels & Growth — 22%
·Unit Economics — 14%  |  Subtraction — 12%
·Audience — 10%  |  Website — 10%  |  Content & Authority — 10%  |  Risks — 10%
Anti-Inflation Caps

Caps prevent score gaming. If a condition is true, the affected dimension is capped — even if the raw score is higher. Active caps show as red lock icons in the scorecard.

No strategy docs uploaded — all sections specificity capped at 6
No validated differentiators — Positioning confidence capped at 6
Estimated close rate (contains ~ or "estimate") — Economics data capped at 7
No CPC data — Economics and Channels confidence capped at 6
Fewer than 3 competitors — Positioning confidence capped at 5
No analytics integration — Execution data capped at 5
Website-only voice (no strategy doc) — Brand confidence capped at 5
No DR data — Brand data capped at 5
No current activities data — Subtraction data capped at 4
No channel strategy levers — Growth confidence capped at 3
High-severity risks unmitigated — Risks confidence capped at 5
Budget mismatch (>2x difference) — Channels data capped at 6
Overall Score Caps

These cap the final overall score regardless of section scores:

Demand validation not run (Keywords tab) — overall capped at 6.5
Demand validation: high-severity strategic revision needed — overall capped at 6.5
No subtraction analysis — overall capped at 7.0
Low audit pass rate (<50% across all diagnostics) — overall capped at 7.0
Audience score below 5.0 — overall capped at 6.0 (weak audience foundation limits entire strategy)
Score Integrity Checks
First-pass warning — if Pass 1 scores above 7.0, a warning badge appears on the scorecard (high first-pass confidence usually means anti-inflation caps failed to activate)
Score jump detection — if any section score jumps more than 1.5 points between passes, it is flagged (likely data changed rather than genuine improvement)
Audit Diagnostics (D0–D7)

Each scored section runs 4–8 quality checks after generation. Audit pass rates feed into the confidence dimension — high pass rates boost it, low rates penalise it.

·D0 Audience — segments have sizing, personas use archetype labels (not fictional names), all verticals addressed or parked
·D1 Unit Economics — numbers are realistic, margins and CAC are internally consistent, sensitivity analysis with 3 scenarios
·D2 Positioning — presents 2-3 directions without auto-selecting, messaging blocked until direction chosen
·D3 Subtraction — activities have clear stop/keep verdicts, not just summaries
·D4 Channel Strategy — three budget tiers (current/growth/optimal), allocation sums to 100%, channel mix has gaps flagged
·D5 Growth Plan (in Channels tab) — timeline items have durations and dependencies, funnel architecture present
·D6 Brand — content pillars and voice rules are vertical-aware with per-segment overlays
·D7 Risks — risks have severity scores and actionable mitigations
Score Thresholds
≥ 7.0 — Strong. Strategy is well-supported with real data and specific to the client.
4.0 – 6.9 — Developing. Some gaps remain — check the gap panel and active caps.
< 4.0 — Weak. Major data or analysis missing. Run enrichments and diagnostics.
✓ To raise a score: check the gap panel for missing inputs (many are auto-resolvable), run diagnostics to improve audit pass rates, and upload strategy documents to lift specificity caps.
Recent Features (March 2026)

Audience Intelligence (Strategy → Audience tab) — D0 diagnostic that runs before all other diagnostics. Identifies audience segments (2-5, with sizing, revenue potential, acquisition difficulty), buying motions (how each segment researches and decides), persona profiles (specific job titles, goals, frustrations, language patterns), purchase triggers (events that move prospects from aware to active), and an objection map (objections tied to segments with counter-messaging and proof). Validation section recommends which segment to prioritise and flags data gaps. Scoring caps: no segments = data capped at 5.0; no buying motions = confidence capped at 6.0; generic personas = specificity capped at 5.0; over-targeted (6+ segments) = overall capped at 6.5.

Positioning Direction (Strategy → Positioning tab) — Founder-aligned positioning hypothesis system at the top of the Positioning tab. Enter 1-4 positioning hypotheses (or scan the strategy document to auto-extract them), then Evaluate Hypotheses to stress-test each against competitor data, differentiators, and demand signals. AI produces per-hypothesis verdict (viable/partially viable/contested) with provability, distinctiveness, market demand, and risk scores + evidence + reframing suggestions. Then recommends 2-3 optimal directions with headlines, rationale, and trade-offs. Strategist selects a direction (or writes a custom one). Selected direction constrains all D2 outputs: positioning angle, value proposition, messaging hierarchy, brand voice, proof strategy. Direction appears in scorecard header. Scoring caps: no hypotheses = data capped at 5.0; evaluated but unselected = confidence capped at 6.5; low provability = specificity capped at 6.0. Compiled strategy document includes "Positioning Direction" and "Audience Intelligence" sections (now 14 sections total).

Strategy Alignment Column (Sitemap) — Coloured dot per page showing alignment with approved strategy: green = page type backed by a high-priority channel lever, yellow = review needed (lever exists but low priority or no lever found), red = cut suggested by subtraction analysis. Hover tooltip shows revenue estimate for non-structural pages: vol x CTR x close_rate x deal_size = $/mo. Data sources: channel_strategy.levers, execution_plan.lever_details.website.architecture_direction, subtraction.current_activities_audit.

Content Pillar Tags (Sitemap) — Blog pages show a blue tag with their assigned content pillar from brand_strategy.content_pillars. Unassigned blog pages show a grey "Unassigned" tag when strategy data exists. Pillars toolbar button AI-assigns all blog pages in one Claude call. Edit mode shows a pillar dropdown for manual assignment. Requires D6 Brand diagnostic to have populated content pillars.

Priority Auto-Suggestion (Sitemap) — Strategy-driven priority suggestions based on growth plan phases and channel lever scores. Toolbar shows count of pages differing from strategy suggestion. Edit mode shows a blue "P1 ✓" / "P2 ✓" accept button per row where suggestion differs. Accept All toolbar button applies all suggestions at once. Structural pages always suggest P1 (home/contact) or P2 (about/utility). Revenue potential calculation replaces hard volume thresholds for niche verticals.

Key Limits & Architecture Notes
·Keyword pool cap: top 300 scored, shown all in Opportunities. Vol "—" = no DataForSEO data for that country — still selectable manually.
·Keyword scoring: log(vol+1) × 100 / max(KD, 5). Threshold: vol ≥ 10 to score above 0. KD of 0 is treated as 5 to avoid divide-by-zero.
·Country selector auto-detects from geo text but saves your manual selection with the project. Always verify before Fetch Volumes.
·SERP Intel: fetches top 3 organic results. Some sites block crawlers (fetch_ok: false) — this is normal. Directives still generated from whatever was fetched.
·Brief versions: last 2 kept. V1 is dropped when V3 is generated. Regenerate Brief = update current draft in-place. Run V2 = push new draft.
·Copy HTML is stored in separate KV keys per page (not in the main project payload) — survives reload. Main payload stores only audit results and approval state.
·Images — image data stored in separate per-slug KV keys (img:{projectId}:{slug}), not the main project payload. Slot metadata (type, source, approval status, briefs) persists across sessions. Base64 image data is stripped from the main save — actual images are in their own KV entries.
·Monthly keyword trends stripped from save payload — reconstructed from DataForSEO on next Fetch Volumes run.
·SERP Intel competitors stripped from save payload — only directives + metadata persists. SERP competitors visible in brief card only until next save.
·KV limit: 25MB max. Monitor console for [save] payload size warnings. Projects with 100+ pages and heavy research data approach this limit.
·Save: auto-saves on every action (debounced). Green indicator top-right confirms saved. Red = failed — check console for payload size or network error.
·Auth: Cloudflare Access (Google SSO). Any @setsail.ca email can authenticate. Projects are user-scoped (u:{email}:project:{id}). Roles (Admin/Strategist/Viewer) and per-project access restrictions managed in Admin panel (top-right dropdown → Admin).
·Token limits: Default 3k. Briefs: 8k. Copy: 3k (default). Schema: 4k. Research: 8k (brand/business) or 6k (others). Strategy diagnostics: 8k. Strategy output: 8k. Server cap: 8192.
·Bulk queue: Cloudflare Queues (setsailos-gen-queue). 2 concurrent, 3 retries, 30s delay on 429. Job status in KV, 24h TTL. Free tier handles ~10k ops/day — 40 briefs ≈ 120 ops.
·Canadian spelling: -ize endings are correct (optimize, organize, specialize). Only flag: color/honour/center/theatre and -yze → -yse (analyze → analyse).
